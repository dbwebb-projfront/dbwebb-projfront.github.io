## Övning: Happy path

**Syftet** med den här övningen är;  
- att ni som grupp ska komma igång med arbetet i denna första längre sprint.
- att ni som grupp ska bli medvetna om och lära er att bryta ner arbete i hanterbara delar och leverera något funktionellt _innan_ ni förfinar det – och sedan medvetet hantera det som eventuellt kan gå snett eller bli fel. Det är ett arbetssätt som är hållbart, särskilt i långa projekt.

---

### Från krav till user story

Kravspecifikationen beskriver *vad* systemet ska göra – inte hur ni ska organisera arbetet.
Det är ert jobb att bryta ned kraven i hanterbara delar innan ni börjar koda. Men hur kommer man igång då?  
Lättare sagt än gjort kanske. Här nedan kommer lite tips med hur ni kan komma igång i gruppen.

En rimlig väg kan se ut så här:

**1. Välj ett krav**
Om ni har svårt att hitta en ingång så är en rekommendation att börja med att se över de krav med hög prioritet. Försök identifiera ett krav som är avgränsat och inte beror på något ni ännu inte byggt. Gör detta tillsammans i gruppen så att ett resonemang leder fram till ett rimligt val. Målet är alltså att hitta ett krav med hög prio och som inte beror på andra krav eller att en kodbas redan finns.

Exempelvis, om det finns ett krav med hög prioritet som säger att _användare ska kunna registrera och autentitisera sig_ — skulle det kunna var ett bra krav att starta med. Det är högt prioriterat, tekniskt oberoende, kräver ingen tidigare data på plats (registrering) och utgör en förutsättning för i stort sett allt annat i systemet. Alltås, inom ett sådant krav är registrering det rimliga första steget: utan ett konto finns inget att logga in med.

Undersök nu API:t för att se vilka routes och endpoints som är relevanta, och hur ni kan lösa det i React (om ni valt ramverket).

**2. Formulera en user story**
En user story sätter kravet i ett användarperspektiv. Utifrån ovan valda krav skulle en story kunna formuleras så här:

*"Som ny användare vill jag kunna skapa ett konto med e-post och lösenord, så att jag kan logga in och komma åt mina filer."*

Den skulle också kunna bli tre user stories;

- *"Som ny användare vill jag kunna skapa ett konto."*. 
- *"Som användare vill jag kunna logga in."*
- *"Som inloggad användare ska jag kunna komma åt mina filer"*

Kanske skulle det kunna bli ännu fler.  
Angående mängden så finns inget _rätt svar_ - dock bör man undvika att de tasks som kopplas till respektive user story inte krockar eller påverkar varandra för mycket - det är inte bra om flera personer implementerar samma funktionalitet samtidigt. Fler user stories kan vara lockande men kan potentiellt ge upphov till problem om man inte planerar noggrant. Å andra sidan kan för stora stories genera för många eller för stora enskilda tasks. En mellanväg är oftast bäst. Det viktigt för ert arbete är att testa er fram och hitta er väg. 

Låt oss välja mellanvägen - vi konstruerar två stycken user stories.  

1. **Registrering** - _"Som ny användare vill jag kunna skapa ett konto med e-post och lösenord, så att jag kan börja använda applikationen."_
2. **Inloggning och utloggning** - _"Som registrerad användare vill jag kunna logga in och ut, så att jag kan komma åt och skydda mina filer."_



**3. Definiera acceptance criteria**
Vi går vidare med Acceptance criteria som ska ge svar på frågan: *vad måste vara sant för att storyn ska anses klar?*. 
För **Registrering** är gruppen överens och kommer fram till följande;  
- Registreringsformuläret visas för icke-autentiserade användare
- En ny användare kan skapa ett konto med giltig e-post och lösenord
- Felaktiga eller ofullständiga uppgifter ger ett begripligt felmeddelande
- Efter lyckad registrering är användaren inloggad och omdirigeras till applikationen

För **Inloggning och utloggning** ges:
- Inloggningsformuläret visas för icke-autentiserade användare
- Korrekt e-post och lösenord leder till autentiserad session
- Felaktiga uppgifter ger ett begripligt felmeddelande
- Inloggad användare kan logga ut och sessionen avslutas

Exakt hur acceptance criterias ska se ut finns inget facit på. Men de ska svara på frågan ovan för att säkerställa att det vi arbetat med är klart.

Om Definition of Done finns på plats i gruppen så är det gruppens generella 'kvalitetskrav' som gäller för all kod ni levererar, oavsett story. DoD gäller då på allt, utöver acceptance criterias.
Exempel på DoD: koden är granskad av minst en annan gruppmedlem, den går att köra lokalt utan fel, och den är mergad till main via pull request. 
Acceptance criteria och DoD kompletterar varandra: en story är klar när båda är uppfyllda. 
Exempelvis kan gruppen ha Code review och merge ska göras på samtliga User Stories - då hade det inte behövts i acceptance criterias (om man inte vill vara extra tydlig förstås).

**4. Bryt ned i tasks**
Nu ska arbetet konkretiseras i utförbara tasks. 
Undvik att skriva exakt alla detaljer — det lämnas till den som utför arbetet att lösa på det sätt de finner bäst. Det som kan vara svårt är att hitta rätt nivå: tasks ska varken vara för grova eller för detaljrika. Sikta på att varje task ryms inom en arbetsdag — på så sätt går arbetet framåt och kodbasen växer steg för steg.

Utifrån ovan och för **Registrering** skapas följande tasks:
- Skapa registreringskomponent (UI)
- Koppla formulär mot autentiserings-API
- Hantera felrespons från API
- Code review och merge

och för **Inloggning och utloggning**;
- Skapa inloggningskomponent (UI)
- Koppla formulär mot autentiserings-API
- Hantera felrespons från API
- Implementera utloggning och rensa session
- Code review och merge

**5. Estimera**
Slutligen, sätt ett tidsestimat per task — tillsammans bildar de ett estimat för hela user storyn.
Ni kommer troligtvis landa lite fel – det är helt i sin ordning och blir bättre med varje sprint.  
Är ni inte överens — diskutera varför. Ofta avslöjar oenighet att tasken är otydlig eller för stor eller att vissa gruppmedlemmar har mer information att delge.

Det finns metoder som kan underlätta estimering i grupp, exempelvis [Planning poker](https://www.atlassian.com/blog/platform/a-brief-overview-of-planning-poker) — ett enkelt sätt att få fram ett gemensamt estimat utan att en person dominerar diskussionen.

### Innan ni börjar koda

Innan gruppen sätter sig och skriver den första raden kod är det värt att stanna upp och säkerställa att ni har en gemensam grund att stå på. 
Det handlar inte om att planera allt i detalj – det handlar om att undvika friktion som annars dyker upp mitt i en sprint.

Några saker som bör vara på plats:

**Teknisk setup**  
Alla i gruppen bör kunna köra projektet lokalt. Det låter självklart, men det är vanligt att en hel dag försvinner för att någon inte har rätt version av Node, eller att miljövariabler saknas. 
Säkerställ gärna att:
- Repot är klonat och projektet hos var och en och startar utan fel för alla
- Ni har börjat undersöka hur ni använder det tillhandahållna API:et (vilka endpoints som finns och vad dom gör)
- Er utvecklingsmiljö i teamet är konsekvent – samma verktyg, samma versioner o.sv.

**Definition of Done**  
Utan en gemensam bild av vad "klar" betyder finns en risk att kod mergas som inte fungerar. Eller att arbeta i olika riktning(ar). 
Bestäm tidigt – gärna nu – vad som gäller för ert team. Behöver koden vara testad och vad menar vi med testad? Code review? Ska den fungera på mobil? Era kriterier behöver inte vara långa, men det är starkt rekommendat att åtminstone en början på dessa finns.

**Vilket krav börjar ni med?**  
Kravspecen har prioriteringar som är satta av en anledning: krav med hög prioritet är ofta tekniskt grundläggande och andra krav är svåra att implementera utan dem. 
Ett rimligt första steg, så här i början, är att identifiera ett krav som:  
- Är relativt avgränsat
- Ger er något konkret och körbart att visa upp
- Inte är beroende av något ni ännu inte byggt

Tänk på att ni inte behöver börja med det mest komplexa – det viktiga är att ni börjar med något som ger hela gruppen en gemensam kodbas att bygga vidare på.

### Happy path

När ni väl börjar implementera ett krav har man ofta ambitionen att lösa allt på en gång – allt ska funka med en gång. Alla eventuella fall som kan leda till fel, alla edge cases (unika fall), alla alla konstiga scenarion man kan komma på - allt ska funka. 
Det är på sätt och vis en instinkt och man menar väl. Men, det kan lätt leda till att lite eller ingenting blir riktigt och fullständigt färdigställt.

*Happy path* är den väg genom ett system som fungerar när allt går rätt till. Ingen saknar behörighet, ingen skriver fel i ett formulär, ingen förlorar uppkopplingen mitt i en save. Det är det scenariot som ni rekommdenderas implementera först och få att fungera fullt ut innan ni tar er an det som kan gå fel.

**Varför?**
- Det ger er något körbart och demonstrerbart tidigt
- Det gör det lättare att resonera om krav och acceptance criteria
- Det är lättare att lägga till felhantering i ett fungerande system än tvärtom

**I praktiken** – välj ett krav och skriv en user story. Definiera happy path: vad händer när en inloggad användare gör precis det som kravspecifikationen beskriver, utan några komplikationer? Implementera det scenariot. Verifiera att det fungerar. Lägg sedan, i mån av tid, till felhantering och edge cases (kanske som egna tasks).

Ett konkret exempel: om ni arbetar med att visa en lista över filer, är happy path att en inloggad användare med befintliga filer ser dem listade. Att hantera en tom lista, ett API-fel eller en användare utan behörighet är viktiga fall – men de är inte happy path, och de behöver inte lösas för att ni ska kunna visa att grundfunktionen fungerar.

Happy path förutsätter att allt fungerar. I verkligheten gör det sällan det – och det är där *sad path* kommer in.

Sad path (ibland kallat *unhappy path* eller *error path*) beskriver vad som händer när något går fel eller när användaren gör något oväntat. Det handlar inte om extrema fel, utan om de situationer som är fullt realistiska i normal användning: ett tomt formulärfält, ett misslyckat API-anrop, en session som löpt ut.

**Exempel sad paths i CoEd:**
- Användaren försöker logga in med fel lösenord
- API:et svarar inte, eller returnerar ett fel
- Användaren försöker öppna en fil den inte har behörighet till
- Nätverkskopplingen tappas mitt i en redigering
- Formulärfält lämnas tomma eller fylls i felaktigt

En närliggande kategori är *edge cases* – situationer som inte nödvändigtvis är fel, men som inträffar i gränslandet av det förväntade. Vad händer om en användare har noll filer? Vad händer om två användare sparar exakt samtidigt? Vad händer om ett filnamn innehåller specialtecken? Edge cases är lätta att missa under implementation eftersom de sällan dyker upp i det normala testflödet – men de är ofta precis de situationer som verkliga användare råkar ut för.

Sad paths och edge cases behöver inte blockera er sprint – men de bör finnas med som tasks, och de bör implementeras *efter* att happy path fungerar. Ett system som hanterar det lyckliga scenariot men kraschar vid det minsta oväntade är inte leveransklart.

**I praktiken** 

efter att ni implementerat och verifierat happy path, gå igenom era acceptance criteria och fråga: vad kan gå fel här, och vad händer i gränsfallen? Varje sådant scenario är ett potentiellt sad path eller edge case. 
Lägg in dem som egna tasks och prioritera dem utefter hur sannolika och allvarliga de är. 
Felmeddelanden som är begripliga för användaren är nästan alltid värt att implementera tidigt – de hjälper er också att felsöka under utvecklingen.
